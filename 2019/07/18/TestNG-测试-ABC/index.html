TestNG 测试 ABC

Posted on 2019-07-18 | In 测试 | 次

前言

单元测试不仅可以检查代码中是否存在缺陷漏洞，也可以降低bug修复、需求变更等引发的代码风险。良好的单元测试对产品开发维护和技术提升有着积极而重大的意义。

如何开始单元测试

1.1 开发环境

IDEA 启用 TestNG 插件

1.2 maven 依赖

1.3 创建测试

  • 类名 alt + enter 创建单元测试

  • xml 文件配置单元测试

1.4 执行测试

  • 测试方法右击执行 run 、 debug

  • maven

1.5 检查覆盖率

TestNG 使用

2.1 特点

• 支持注解
• 支持多线程和线程安全
• 数据驱动
• 灵活的运行时配置
• 支持分组
• 支持依赖测试方法、并行测试、负载测试、局部故障
• 丰富的插件 API

2.2 基础概念

  • 套件 Suit由一个 XML 文件所表示。它能够包含一个或者多个测试， 标记来定义
  • 组 Group TestNG 可以将各个 method 存放在不同的 group 里面，然后运行的时候可以指定要运行的 group

2.3 常用注解

  • @Test 测试注解
  • @BeforeTest 在测试之前执行
  • @AfterTest 在测试之后执行
  • @BeforeClass 在测试类执行之前执行
  • @AfterClass 在测试类执行之后执行
  • @BeforeMethod 在测试方法执行之前执行
  • @AfterMethod在 测试方法执行之后执行
  • @BeforeSuite 在测试套件执行之前执行
  • @AfterSuite 在测试套件执行之后执行
  • @BeforeGroups 在测试组执行之前执行
  • @AfterGroups 在测试组执行之后执行
  • @Test(enable=false) 忽略测试
  • @Test(expectedExceptions = Exception.class) 设置异常
  • @Test(timeout = 1000) 设置超时时间

2.4 测试类型

2.4.1 时间测试

设置单元测试执行的超时时间

2.4.2 异常测试

指定测试中抛出的异常

2.4.3 忽略测试

忽略指定的测试

2.4.5 套件测试

将多个测试组合成一个模块并执行

2.4.6 组测试

将各个 method 存放在不同的 group 里面，然后运行的时候可以指定要运行的 group

2.4.7 参数化测试

指定传多种参数值，验证接口对多种不同参数的处理是否正确，有两种方式注解@DataProvider、xml

  • xml 配置参数

  • DataProvider 配置参数

2.4.8 依赖测试

在执行的测试之前需要执行的另一测试。如果依赖的测试出现错误，所有的子测试都被忽略，且不被标记为失败。

2.4.8 性能测试

通过多个线程并发调用测试接口来实现性能测试

  • invocationCount 执行次数
  • threadPoolSize 线程数
  • timeOut 超时时间
  • invocationTimeOut 总耗时时间

单元测试规范

3.1 命名规范

  • 目录结构：项目目录结构下的单元测试目录test
  • 包名：被测试类包名
  • 类名：被测试类名 + Test
  • 方法名：test + 被测试方法名 + 场景

3.2 测试方向

3.2.1 模拟接口测试

  1. 模块输入参数数量、属性、类型等是否一致
  2. 传到被调用模块的实参的数量、属性、类型是否与被调用模块形参一致
  3. 引用内部函数时，实参的顺序和数目是否正确
  4. 是否引用了与当前入口无关的参数
  5. 用于输入的变量有没有改变
  6. 不同模块的全局变量的定义是否一致
  7. 限制条件是否以形参的形式传递
  8. 是否及时释放资源

3.2.2 局部数据结构测试

  1. 不正确或不一致的数据类型说明
  2. 使用未初始化的变量
  3. 错误的初始值或错误的默认值
  4. 数据类型不一致
  5. 全局变量影响
  6. 数组越界

3.2.3 路径测试

  1. 不同数据类型的比较
  2. 不正确的逻辑运算符或优先次序
  3. 浮点运算精度问题
  4. 不正常的或不存在的循环中的条件
  5. 无法跳出/终止循环
  6. 错误修改循环变量

3.2.4 错误处理测试

  1. 是否能够对错误定位
  2. 显示与实际的错误是否一致
  3. 是否正确处理错误条件
  4. 是否可以干预错误条件

3.2.5 边界测试

  1. 普通合法数据是否正确处理
  2. 普通非法数据是否正确处理
  3. 边界内最接近边界的（合法）数据是否正确处理
  4. 边界外最接近边界的（非法）数据是否正确处理等
  5. 在n次循环的第0次、第1次、第n次是否有错误

3.3 测试覆盖率

  1. 语句覆盖达到100%
    指被测单元中每条可执行语句都被测试用例所覆盖
  2. 分支覆盖达到100%
    指分支语句取true和false各一次。分支覆盖使这些分支产生的输出都得到验证，提高测试的充分性。
  3. 覆盖错误处理路径
  4. 单元的软件特性覆盖，包括功能、性能、属性、设计约束、状态数目、分支的行数等
  5. 排除异常的输入。

单元测试使用总结

  1. 单元测试努力维持在Class级别，保证易执行
  2. 努力全自动
  3. 评估分析测试结果
  4. 保持独立
  5. 不测试私有方法，降低复杂度
  6. 重要方法100%覆盖
  7. 尽可能提高测试覆盖率
  8. bug可重现
  9. 单元测试无法保证代码正确(抛锚说明汽车肯定有问题，但是能跑不代表汽车没故障)
